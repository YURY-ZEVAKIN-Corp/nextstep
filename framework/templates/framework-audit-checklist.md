# Framework Audit Checklist

Use this checklist to audit the framework itself rather than any one project.

## 1. Structure Consistency

- [ ] The documented framework structure matches the actual folder and file model.
- [ ] Portfolio-level and project-level responsibilities are clearly separated.
- [ ] Templates align with the operating model described in the main framework guide.

## 2. Rule Consistency

- [ ] Framework rules do not contradict each other.
- [ ] Artifact lifecycle rules are ordered logically.
- [ ] Status definitions are usable and not conflicting across artifacts.
- [ ] Required artifacts are referenced consistently in all guidance.
- [ ] Framework rules remain consistent with the declared anchor models.

## 3. Conflict Search

- [ ] Checked for conflicts between standards, templates, and process descriptions.
- [ ] Checked for duplicate rules stated with different meanings.
- [ ] Checked for missing storage locations for required artifacts.
- [ ] Checked for examples that violate the framework rules.

## 4. Traceability Consistency

- [ ] Backlog, requirements, decisions, implementation, testing, and release flow is coherent.
- [ ] Decision governance is traceable through registers and linked artifacts.
- [ ] Technology stack governance is traceable through decision records.
- [ ] Audit governance is traceable through schedule, logs, and checklists.

## 5. Audit System Consistency

- [ ] Project audit, portfolio audit, all-projects audit, and framework audit are clearly differentiated.
- [ ] Each audit type has a clear purpose, cadence, checklist, and output log.
- [ ] Audit rules do not conflict with delivery rules.

## 6. Maintainability

- [ ] The framework is still understandable without hidden assumptions.
- [ ] New projects can adopt the framework without ambiguity.
- [ ] The framework can evolve without duplicating governance logic excessively.
- [ ] Tailoring guidance is clear for small, medium, and high-governance projects.

## 7. Audit Outcome

- [ ] Consistency findings are recorded.
- [ ] Rule conflicts are listed explicitly.
- [ ] Corrective actions are assigned.
- [ ] Next framework audit date is scheduled.

## Result

Mark the audit:

- `Pass` if the framework is internally coherent and operationally usable.
- `Warning` if some inconsistencies exist but do not break governance.
- `Fail` if the framework contains major contradictions, missing controls, or conflicting rules.
