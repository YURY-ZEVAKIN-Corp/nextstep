# Requirement Checklist

Use this checklist for every new requirement before it is marked `Ready`.

## Clarity

- [ ] The problem statement is specific and understandable.
- [ ] The requirement describes an outcome, not a vague intention.
- [ ] The affected users, systems, or stakeholders are identified.

## Value

- [ ] The business or user value is explicit.
- [ ] A business value metric is calculated.
- [ ] The metric inputs and assumptions are documented.
- [ ] The reason this work matters now is documented.

## Scope

- [ ] The scope is clear enough to estimate and design.
- [ ] Out-of-scope items are listed.
- [ ] Dependencies and assumptions are recorded.

## Testability

- [ ] Acceptance criteria are concrete and verifiable.
- [ ] Success can be validated through testing or observation.
- [ ] Non-functional requirements are stated if relevant.

## Readiness

- [ ] Open questions are identified.
- [ ] The requirement has an owner.
- [ ] The requirement is ready to be transformed into a solution.

## Decision Rule

Mark the requirement:

- `Pass` if all critical items are satisfied or have an explicit follow-up.
- `Fail` if the requirement is too ambiguous, not testable, or missing key scope/value information.
