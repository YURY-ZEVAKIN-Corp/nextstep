# Audit Model

This document defines the audit taxonomy for the framework.

## 1. Project Audit

- Purpose: review one project for compliance with the framework
- Scope: one project
- Cadence: every 2 weeks
- Checklist: `templates/audit-checklist.md`
- Output: `projects/<project-key>/audit-log.md`
- Owner: project owner or delivery lead

## 2. Portfolio Audit

- Purpose: review portfolio governance controls and shared management artifacts
- Scope: portfolio-level controls such as standards, schedules, shared decisions, dependency visibility, and reporting discipline
- Cadence: monthly
- Checklist: `templates/audit-checklist.md`
- Output: `portfolio/audit-log.md`
- Owner: PMO, portfolio owner, or project lead

## 3. All Projects High-Level Audit

- Purpose: review the overall health of all active projects at summary level
- Scope: all active projects, using a high-level governance and delivery health pass
- Cadence: monthly or quarterly
- Checklist: `templates/all-projects-audit-checklist.md`
- Output: `portfolio/all-projects-audit-log.md`
- Owner: PMO or portfolio owner

## 4. Framework Audit

- Purpose: review the framework itself for consistency, missing controls, and rule conflicts
- Scope: framework rules, templates, examples, standards, and governance model
- Cadence: monthly or after major framework changes
- Checklist: `templates/framework-audit-checklist.md`
- Output: `portfolio/framework-audit-log.md`
- Owner: framework owner, PMO, or project lead

## Governance Rule

Every audit type must have:

- a clear purpose;
- a defined scope;
- a cadence;
- a checklist;
- an output log;
- an accountable owner.
